# Teaching methods

## Socratic

- Ask for their current mental model first.
- Prefer "what happens if X?" over "do Y".
- When wrong, ask what evidence would change their mind.

## Deliberate practice

- Isolate one weak subskill per session.
- Tight feedback loop (<15 minutes between attempt and review).
- Increase difficulty only after clean success.

## Progressive hints

Orient → Concept → Skeleton → Narrow next edit → Partial example.

## Code reading gym

1. Map entrypoints and data flow without changing code.
2. Predict behavior; verify with tests or small runs (learner runs).
3. Summarize invariants in their own words.
4. Only then propose a change design.

## Design studio

1. Constraints & non-goals
2. 2–3 options with trade-offs
3. Choose and write interface/types first
4. Test list before implementation
5. Implement (learner) in thin vertical slices

## Debugging clinic

1. Reproduce
2. Hypothesize (ranked)
3. Design the cheapest disproof
4. Observe
5. Fix with a regression test

## ML engineering lab

Baseline → metric on a frozen split → simple model → error analysis → complexity only if justified.

## Research seminar

Paper: question, method, evidence, threats to validity, what you would ablate next.

## Feynman close

End with: "Explain it so a sharp junior can implement it tomorrow."

## Motivation without fluff

- Specific praise ("your error handling covered the timeout path")
- Specific next bar ("next: make the API idempotent")
- Never inflate XP or levels
